# Surface DEM Benchmark Design

Build a decode-only benchmark on top of the existing rotated surface-code circuits from `PyMatching/benchmarks/surface_codes/`, but stop the timed comparison at the DEM boundary. For each selected circuit, the benchmark will load the `.stim` file once, derive a decomposed DEM once, and compile a detector sampler with a fixed seed. It will then sample a fixed batch of syndromes and observable flips exactly once before timing begins. The measured section compares only `PyMatching.decode_batch` and `rmatching.decode_batch` on that fixed DEM plus fixed syndrome batch.

The first version will include two representative graph sizes, `d=5` and `d=17`, both at physical error rate `p=0.001`. This keeps the suite small enough to run quickly while still moving beyond the toy DEMs from the minimal microbenchmark. Choosing a low error rate is intentional: sparse syndromes exercise the case where the graph is large but each shot touches relatively few detector nodes, which is the regime where the touched-node reset optimization should matter most.

Implementation will stay parallel to the existing benchmark code instead of extending `run_minimal_benchmark.py` or `run_benchmark.py`. A dedicated case module will define the selected `.stim` inputs, derive fixed DEM text, and generate fixed syndrome batches using deterministic sampling. A dedicated Python driver will reuse the existing timing and mismatch-summary patterns, but write separate artifacts so toy-case results, full-pipeline results, and decode-only surface-code results remain isolated. The output should be enough to answer two questions directly: whether predictions still match PyMatching exactly on these larger graphs, and whether decode time narrows once reset work scales with touched nodes instead of total nodes.
